SECTION 2 — THREE LAYER ARCHITECTURE
================================================================================

────────────────────────────────────────────────────────────
LAYER 1 — MONITORING DAEMON (Rust)
────────────────────────────────────────────────────────────

Always running. Always watching. Does not care if the user is present.

RESPONSIBILITIES:
    — Poll all data sources on defined intervals
    — Parse and normalize incoming data into standard record format
    — Run lightweight classification via Ollama round robin
    — Detect anomalies against learned baselines (Welford algorithm)
    — Detect silences — what didn't arrive that should have
    — Generate QUERY threads when patterns trigger thresholds
    — Write all findings to memory store with full provenance
    — Write checkpoints on defined triggers
    — Fire alerts when escalation thresholds are crossed
    — Log all downtime gaps as monitoring gap records

DOES NOT:
    — Talk to the user
    — Make conclusions
    — Run deep LLM reasoning
    — Attempt crash recovery beyond last checkpoint

DATA SOURCES (WorldMonitor fork + extensions):
    — 100+ RSS feeds (geopolitics, defense, energy, tech)
    — GDELT (global event database)
    — ACLED (armed conflict location and event data)
    — OpenSky (military / civilian flight ADS-B)
    — VesselFinder AIS (naval / maritime tracking)
    — USGS (seismic)
    — NASA FIRMS (satellite fire detection / VIIRS)
    — NASA EONET (natural events)
    — Cloudflare Radar (internet outages / traffic anomalies)
    — FRED (Federal Reserve economic data)
    — Polymarket (prediction markets as leading indicators)
    — Additional free API keys as acquired

CHECKPOINT TRIGGERS:
    — User-initiated request received
    — Significant data write to memory (alert threshold crossed)
    — Defined time interval (TBD — likely 15-30 minutes)

CHECKPOINT DOES NOT TRIGGER ON:
    — Crash
    — Unplanned shutdown

ON RESTART:
    — Load last checkpoint
    — Resume from that state only
    — Log gap between last checkpoint and restart as monitoring gap
    — Do not attempt to reconstruct gap period
    — Treat gap as potential silence data point

────────────────────────────────────────────────────────────
LAYER 2 — MEMORY / KNOWLEDGE STORE
────────────────────────────────────────────────────────────

The substrate. Everything reads from it. Everything writes to it.
Not a layer the user interacts with directly.
The long-term memory of the investigator.

TWO COMPONENTS:

POSTGRESQL — structured memory
    Facts, timestamps, provenance chains, tier assignments,
    source records, QUERY threads, alert history,
    checkpoint state, monitoring gap log,
    entity registry, relationship edges (typed + weighted),
    silence records, decay tracking
    Connection pooling via PgBouncer — prevents connection bloat

CHROMADB — vector memory
    Semantic embeddings of all confirmed facts
    Pattern library (shapes of past silences and convergences)
    Enables plain-language semantic search across all memory
    Relationship proximity mapping between entities
    Used by research layer and conversation layer for context retrieval
    Runs with memory-mapped files — does not load full index into RAM

POSTGRES / CHROMADB SYNC — CRITICAL ENGINEERING PROBLEM:

    THE FAILURE MODE:
    A fact decays in Postgres but its embedding still ranks highly in Chroma.
    Result: semantically relevant but epistemically weak retrieval.
    Trust-destroying if undetected.

    SOLUTION — TWO STAGE, SIMPLE FIRST:

    Stage 1 — Confidence-weighted retrieval (implemented immediately):
        Every ChromaDB retrieval gets confidence-weighted at query time.
        Semantic score multiplied by current Postgres confidence value.
        Low-confidence facts cannot rank highly regardless of semantic match.

        final_score = semantic_similarity * confidence_value * (1 - decay_factor)

    Stage 2 — Embedding refresh (implemented when system is stable):
        Scheduled job monitors confidence values in Postgres.
        When confidence drops below threshold — embedding metadata updated in Chroma.
        Chroma metadata filtering excludes low-confidence embeddings by default.
        Explicitly requested low-confidence retrieval still possible (labeled).

HARDWARE RAM ALLOCATION (20GB total):
    PostgreSQL          —  2-4GB under load
    ChromaDB            —  1-2GB (memory-mapped)
    Rust daemon         —  ~200MB
    Celery workers x4   —  ~500MB each = 2GB
    Redis (broker)      —  ~200MB
    Tauri UI            —  ~300MB
    Ollama (one model)  —  4-6GB (one loaded at a time — not six local)
    OS overhead         —  2-3GB
    ─────────────────────────────────
    Conservative total  —  ~12-18GB
    Headroom            —  2-8GB

    DISCIPLINE REQUIRED:
    One Ollama model loaded at a time — six APIs are remote endpoints, not
    six local processes. This is non-negotiable for RAM management.
    If ceiling is hit — ChromaDB moves to cheap persistent VPS first.
    Everything else stays local.

TIME DECAY:
    — All information decays uniformly over time
    — Rate: TBD (to be calibrated during build phase)
    — Decay does not delete — it reduces confidence value
    — Reconfirmation by new independent source resets decay clock
    — Decayed facts remain in memory, labeled as aged
    — High-tier confirmed facts decay slower (rate multiplier TBD)

────────────────────────────────────────────────────────────
LAYER 3 — RESEARCH + CONVERSATION (Python)
────────────────────────────────────────────────────────────

CONCURRENCY MODEL — REQUIRED, NOT OPTIONAL:
    asyncio + aiohttp for all external calls
    Nothing blocks. Every LLM call, API hit, and memory write
    runs concurrently where possible.

    Task queue: Celery + Redis as message broker
    Each QUERY thread becomes a Celery task
    Tasks run in parallel up to worker limit
    Worker count: capped at 4 (RAM constraint before CPU)
    i7-8700 has 12 threads — worker ceiling is RAM, not CPU

    If Python layer still bottlenecks under load:
    Rust daemon absorbs lightweight orchestration
    Python handles only LLM-dependent tasks
    This is the architectural escape hatch — built in from start

RESEARCH SUBLAYER:
    Triggered by: monitoring daemon QUERY / user question
    — Checks memory first — what is already known
    — If memory insufficient — goes external
    — Runs recursive investigation loop via async task queue
    — Generates sub-QUERYs from findings
    — Hard depth limit per thread (TBD — likely 5-7 recursive levels)
    — Writes all findings to memory with provenance chain
    — Escalates to alert if silence persists past threshold
    — Uses Claude API for complex reasoning
    — Uses Ollama round robin for routine sub-tasks

QUERY EXPLOSION CONTROLS — ALL THREE REQUIRED:
    1. DEPTH LIMIT
       Hard ceiling per thread — 5-7 levels maximum
       Non-negotiable. Enforced at task spawn.

    2. PRIORITY WEIGHTING
       Every QUERY gets significance score at spawn time
       Low significance → fewer worker slots
       High significance → priority queue access
       Queue is weighted — not FIFO

    3. KILL CONDITIONS
       Sub-query generates no new information after N attempts → terminate branch
       Sub-query duplicates existing open QUERY → merge, do not spawn twin
       Deduplication check runs before every spawn — most important control
       Total open threads exceed system limit → new QUERYs queue, do not spawn
       User can manually kill any thread from Tauri UI

CONVERSATION SUBLAYER:
    Triggered by: user
    — Plain language input
    — Semantic search across memory (ChromaDB)
    — Routes to research layer if memory insufficient
    — Returns answer with tier label and provenance
    — Can discuss, challenge, redirect investigations
    — Surfaces what daemon found while user was absent
    — Hard rule: can only assert what exists in memory with a source
    — Opinion and synthesis labeled explicitly as such
    — Cannot invent citations under any condition

SECTION 3 — PRIMARY DATA TYPES
================================================================================

Four types. All information entering or generated by the system
is assigned exactly one type at intake. Type can be upgraded or
downgraded as evidence accumulates. Downgrades are permanent records
— the system remembers when something was demoted and why.

────────────────────────────────────────────────────
TYPE 1 — VERIFIED
────────────────────────────────────────────────────
Definition:
    Confirmed by multiple independent sources across
    different domains with no detectable coordination.
    Cross-domain confirmation required — same-domain
    confirmation alone is insufficient.

Yetzirah mapping: 3 Mothers — axiomatic foundation
Confidence floor: HIGH
Decay rate: Standard (slowest of the four types)

Upgrade path: CONTESTED → VERIFIED
    Requires: cross-domain independent confirmation
    Minimum sources: 3 (TBD — to be calibrated)
    Coordination check: must pass

────────────────────────────────────────────────────
TYPE 2 — CONTESTED
────────────────────────────────────────────────────
Definition:
    Real phenomenon with credible sourcing.
    Disputed interpretation or incomplete evidence.
    Could resolve either direction.

Yetzirah mapping: 7 Doubles — dual nature, could resolve either way
Confidence floor: MEDIUM
Decay rate: Standard

Upgrade path: SPECULATIVE → CONTESTED
    Requires: credible sourcing, no primary source contradiction
Downgrade path: CONTESTED → SPECULATIVE
    Requires: primary source contradiction or source discrediting

────────────────────────────────────────────────────
TYPE 3 — SPECULATIVE
────────────────────────────────────────────────────
Definition:
    Unverified. Circulating. No archival confirmation.
    Present in the system because absence of confirmation
    is not confirmation of absence.

Yetzirah mapping: 12 Simples — in motion, unresolved
Confidence floor: LOW
Decay rate: Standard (fastest effective decay due to low floor)

Note: SPECULATIVE items are never surfaced as findings.
They are held in memory as investigation seeds only.

────────────────────────────────────────────────────
TYPE 4 — QUERY
────────────────────────────────────────────────────
Definition:
    Internally generated investigation thread.
    Spawned by the system from its own findings.
    Not external data. A question the system asks itself.

Yetzirah mapping: The recursive loop — Yesod to Tiphareth
Confidence floor: N/A — not a fact, a question
Decay rate: Does not decay — persists until resolved or closed

States:
    OPEN        — actively being investigated
    ESCALATED   — silence persisted past threshold / alert fired
    RESOLVED    — closed with findings (findings carry their own tier)
    CLOSED      — closed without findings (absence documented)

QUERY threads are never dropped.
A QUERY closed without findings is itself a finding.
The persistence of silence under active investigation
is logged as a SILENCE RECORD (see Section 4).

SCHEMA A — INFORMATION RECORD
The fundamental unit. Every piece of data entering the system.
────────────────────────────────────────────────────

information_record {

    -- IDENTITY
    id                  UUID            PRIMARY KEY
    created_at          TIMESTAMP       NOT NULL
    ingested_at         TIMESTAMP       NOT NULL        -- when daemon received it

    -- CONTENT
    raw_content         TEXT            NOT NULL        -- exactly as received
    normalized_content  TEXT                            -- cleaned, standardized
    summary             TEXT                            -- LLM-generated summary
    language            VARCHAR(10)                     -- ISO 639-1
    translated_content  TEXT                            -- if translation occurred

    -- SOURCE
    source_id           UUID            FK → source_record
    source_url          TEXT
    source_domain       VARCHAR(100)
    publication_time    TIMESTAMP                       -- when source published it
    retrieval_method    VARCHAR(50)     -- RSS / API / scrape / manual / internal

    -- CLASSIFICATION
    data_type           ENUM            -- VERIFIED / CONTESTED / SPECULATIVE / QUERY
    previous_type       ENUM                            -- if type changed
    type_changed_at     TIMESTAMP
    type_change_reason  TEXT

    -- DOMAIN TAGS
    domains             TEXT[]          -- geopolitical / financial / military /
                                        -- legal / social / environmental /
                                        -- technological / spiritual / other

    -- GEOGRAPHIC
    geo_entities        TEXT[]          -- countries, regions, cities mentioned
    geo_coordinates     POINT[]         -- if extractable
    geo_confidence      FLOAT           -- 0.0 to 1.0

    -- ENTITY EXTRACTION
    entities            JSONB           -- persons, organizations, institutions
                                        -- {name, type, confidence, role}

    -- CONFIDENCE & DECAY
    confidence_value    FLOAT           -- 0.0 to 1.0
    confidence_floor    FLOAT           -- minimum after full decay
    decay_rate          FLOAT           -- per-day reduction
    last_confirmed_at   TIMESTAMP       -- last independent reconfirmation
    confirmation_count  INTEGER         -- number of independent confirmations

    -- PROVENANCE CHAIN
    source_tier         INTEGER         -- 1 (wire) through 4 (blog/aggregator)
    propaganda_risk     FLOAT           -- 0.0 to 1.0
    state_affiliated    BOOLEAN
    affiliated_state    VARCHAR(100)    -- if state_affiliated = true
    coordination_flag   BOOLEAN         -- potential coordination detected
    coordination_notes  TEXT

    -- CROSS-REFERENCES
    related_records     UUID[]          -- other records this connects to
    confirms_records    UUID[]          -- records this independently confirms
    contradicts_records UUID[]          -- records this conflicts with
    spawned_query       UUID            FK → query_record (if this created a QUERY)

    -- CONTEXT
    displaced_by        UUID            FK → information_record
                                        -- if a simultaneous loud story
                                        -- broke at same time as this silence
    displacement_notes  TEXT

    -- INTERNAL
    classified_by       VARCHAR(50)     -- ollama-[model] / claude / keyword / manual
    classification_confidence FLOAT
    embedding_id        VARCHAR(100)    FK → ChromaDB vector ID
    raw_hash            VARCHAR(64)     -- SHA256 of raw_content (dedup)
}

SECTION 5 — OLLAMA ROUND ROBIN POOL
================================================================================

Six endpoints. Rotating. Health-checked before each call.
Failed endpoints go to back of queue. Cooldown before retry.

pool_config {
    endpoints           TEXT[]          -- six Ollama API URLs
    current_index       INTEGER         -- round robin pointer
    health_check_timeout INTEGER        -- ms before marking endpoint unhealthy
    cooldown_period     INTEGER         -- seconds before retrying failed endpoint
    retry_attempts      INTEGER         -- per request before falling back
    fallback            ENUM            -- CLAUDE_API / OPENROUTER / SKIP
}

ROUTING LOGIC:
    Monitoring classification      →  Ollama pool (always)
    Routine research sub-tasks     →  Ollama pool (preferred)
    Complex reasoning              →  Claude API (preferred) / Ollama fallback
    Conversation                   →  Claude API (preferred) / Ollama fallback
    Model consensus (future)       →  two different Ollama endpoints
                                      Agreement = higher confidence
                                      Disagreement = flag for human review

MODEL SELECTION: flagged for build phase
    Decision deferred until hardware benchmarking under load.

SECTION 9 — ENGINEERING RISKS & MITIGATIONS
================================================================================

RISK 1 — PYTHON LAYER BOTTLENECK
    Risk: Research layer doing recursive queries, external calls, LLM reasoning,
          and memory writes becomes choke point under load.
    Mitigation:
        asyncio + aiohttp — nothing blocks, all I/O concurrent
        Celery + Redis task queue — QUERY threads run as parallel workers
        Worker ceiling: 4 (RAM-constrained, not CPU-constrained)
        Escape hatch: Rust daemon absorbs orchestration if Python still bottlenecks
        Only LLM-dependent tasks remain in Python at that point
    Status: MITIGATED — architecture designed around this from start

RISK 2 — CHROMADB / POSTGRES SYNC DRIFT
    Risk: Fact decays in Postgres but embedding still ranks highly in Chroma.
          Semantically relevant but epistemically weak retrieval corrupts output.
    Mitigation:
        Stage 1: Confidence-weighted retrieval at query time (immediate)
            final_score = semantic_similarity * confidence_value * (1 - decay_factor)
        Stage 2: Scheduled embedding refresh with metadata filtering (post-stable)
    Status: MITIGATED — two-stage solution defined

RISK 3 — COORDINATION DETECTION ACCURACY
    Risk: "Coordinated silence" becomes pattern projection instead of detection.
          System sees coordination where there is none.
    Mitigation:
        Phase 1 — statistics only, no ML:
            Timing analysis from ingestion timestamps (already computable)
            Welford z-score on story-type absence vs baseline
            Boundary clarity score (ragged vs clean silence boundary)
            Political alignment coverage score (0.0 to 1.0)
        Phase 2 — behavioral clustering ML (future, does not block Phase 1)
        All coordination findings labeled CONTESTED until cross-domain confirmed
        System never asserts coordination — it reports the measurable fingerprint
        and labels it. Human concludes.
    Status: PARTIALLY MITIGATED — Phase 1 defined, Phase 2 flagged

RISK 4 — RECURSIVE QUERY EXPLOSION
    Risk: Sub-queries and recursive depth produce exponential thread growth.
          System drowns in its own investigations.
    Mitigation:
        Hard depth limit: 5-7 levels, enforced at spawn
        Priority weighting: significance score gates worker slot access
        Kill conditions (all three enforced):
            No new information after N attempts → terminate branch
            Duplicate question detected → merge to existing thread
            System thread limit exceeded → queue, do not spawn
        Deduplication check before every spawn — most critical control
        Manual kill available from Tauri UI at any time
    Status: MITIGATED — three controls defined and required

RISK 5 — HARDWARE CEILING
    Risk: i7-8700 / 20GB RAM hits memory and latency ceiling under full load.
    Mitigation:
        RAM allocation documented and budgeted (see Section 2, Layer 2)
        Conservative total: 12-18GB — 2-8GB headroom
        One Ollama model at a time — six remote APIs, not six local processes
        Celery workers capped at 4
        ChromaDB memory-mapped — does not load full index into RAM
        PgBouncer connection pooling on Postgres
        First thing to move off-box if ceiling hit: ChromaDB → cheap VPS
        Everything else stays local
    Status: MITIGATED — within hardware limits with discipline
