## Core Data Types

Four types. All information entering or generated by the system is assigned exactly one type at intake. Type can be upgraded or downgraded as evidence accumulates. Downgrades are permanent records.

| Type | Description | Confidence | Decay |
|------|-------------|------------|-------|
| **VERIFIED** | Confirmed by multiple independent cross-domain sources with no detectable coordination | HIGH | Slowest |
| **CONTESTED** | Real phenomenon, credible sourcing, disputed interpretation or incomplete evidence | MEDIUM | Standard |
| **SPECULATIVE** | Unverified, circulating, no archival confirmation. Held as investigation seeds only | LOW | Fastest effective |
| **QUERY** | Internally generated investigation thread. A question the system asks itself | N/A | Does not decay |

QUERY states: `OPEN` → `ESCALATED` → `RESOLVED` or `CLOSED`

A QUERY closed without findings is itself a finding. The persistence of silence under active investigation is logged as a silence record.

## Database Schemas

Eight core schemas define all data structures:

| Schema | Purpose |
|--------|---------|
| **information_record** | The fundamental unit. Every piece of data entering the system. Source, classification, confidence, decay, provenance, cross-references. |
| **source_record** | Every source the system knows. Trust score (computed, never assigned), track record, circuit breaker state. |
| **query_record** | Internally generated investigation threads. Origin, depth, state, findings, silence tracking. |
| **silence_record** | Absence as primary data type. Expected content, detection method, silence typing (organic/coordinated), boundary clarity, displacement events. |
| **entity_record** | People, organizations, institutions. Canonical names, aliases, mention velocity, anomaly detection. |
| **entity_relationship** | Typed edges between entities. CONTROLS, FUNDS, COORDINATES_WITH, PUBLICLY_OPPOSES, etc. Evidence-backed, confidence-scored. |
| **checkpoint_record** | System state preservation. Open queries, daemon state, memory sync markers, integrity checksums. |
| **monitoring_gap_record** | Downtime logged as data. Gap duration, cause, active queries at gap time, silence treatment. |

Full schema definitions: [`docs/schemas/database-schemas.md`](docs/schemas/database-schemas.md)

## Silence Detection

EMET treats absence as a primary data type. Silence is not nothing — it is measurable, classifiable, and in many cases more significant than noise.

**Silence fingerprint analysis:**
- **Boundary clarity** — 0.0 (ragged/organic) to 1.0 (clean/coordinated). Clean boundaries signal coordination.
- **Political alignment coverage** — 0.0 to 1.0. Universal silence across all political alignments is a high-severity signal.
- **Displacement events** — A loud story breaking simultaneously with a silence event. Confirmed displacement elevates alert severity.

An event that generates universal media silence but spikes the GCP network is the strongest possible signal of coordinated narrative suppression.

## Hardware Constraints

| Resource | Allocation |
|----------|-----------|
| PostgreSQL | 2-4 GB under load |
| ChromaDB | 1-2 GB (memory-mapped) |
| Rust daemon | ~200 MB |
| Celery workers x4 | ~500 MB each = 2 GB |
| Redis (broker) | ~200 MB |
| Tauri UI | ~300 MB |
| Ollama (one model) | 4-6 GB (one loaded at a time) |
| OS overhead | 2-3 GB |
| **Conservative total** | **~12-18 GB** |
| **Headroom** | **2-8 GB** |

One Ollama model loaded at a time — six APIs are remote endpoints, not six local processes. Non-negotiable for RAM management. If ceiling is hit — ChromaDB moves to cheap persistent VPS first. Everything else stays local.
